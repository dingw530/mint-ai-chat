## 背景与目标
- **业务背景**：AI Chat V1.5 已具备 MCP 工具集成、自定义 Agent、记忆机制和智能路由。但当前工具调用循环仅支持单次迭代（AI 调用 → 工具执行 → AI 调用 → 结束），无法链式调用多个工具，也无法展示 AI 的推理过程。
- **当前问题**：
  1. `aiProxy.ts` 中工具循环硬编码为 1 次迭代（第 245 行 `toolCalls: null`），AI 拿到工具结果后无法再次调用工具，无法完成多步任务（如"查天气→根据天气推荐活动→把结果写入文件"）。
  2. 工具调用过程对用户不可见，用户看到的是"静默等待"，不知道 AI 在"思考"还是"调用工具"。
  3. 工具调用失败无重试机制，一次失败直接透传错误给 AI，影响体验。
  4. Agent 之间无协作，"锁定 Agent"后只能使用该 Agent 的能力，无法委派子任务给其他专业 Agent。
  5. 路由系统仅做单次"选人"决策，无法对复杂任务进行拆解和编排。
- **成功标准**：
  - AI 可链式调用最多 5 次工具（可配置），完成多步任务。
  - 用户实时看到 AI 的推理过程（思考→调用→观察→思考…），每个步骤可视化。
  - 工具调用失败自动重试（最多 5 次，可配置），不在前端暴露原始错误。
  - 编排 Agent 能拆解复杂任务，委派给 Worker Agent 执行。
  - 所有新功能向后兼容，存量对话不受影响。

## 用户与场景
- **目标用户**：AI Chat 用户（技术用户通过 MCP 工具完成多步操作；普通用户享受更透明的 AI 推理过程）
- **典型场景**：
  - 用户对一个绑定了文件 + GitHub + 搜索 MCP 的 Agent 说："搜索最近的 AI 论文，把摘要写到 `~/notes/ai-papers.md`，然后创建一个 GitHub Issue 总结"。AI 依次执行 搜索→写入文件→创建 Issue，用户实时看到每一步。
  - 用户问："北京下周天气如何？适合户外活动吗？"编排 Agent 拆解为"查北京下周天气"→"分析是否适合户外"，分别委派给天气 Agent 和通用助手，最终汇总回答。
  - 用户看到 AI 在"思考中…"→"正在查询天气…"→"得到结果：北京下周晴，15-25°C"→"分析户外活动适宜性…"→"最终回答"，每一步都清晰可见。
  - 工具调用因网络问题失败，AI 自动重试 2 次后成功获取数据，用户无感知。

## 用户故事
- **US-030**：作为用户，我希望 AI 能连续调用多个工具完成复杂任务（如搜索→写入文件→创建 Issue），从而无需手动执行中间步骤。
- **US-031**：作为用户，我希望在对话中实时看到 AI 的思考过程（ Thought ）、正在调用的工具（Action）和工具返回结果（Observation），从而理解 AI 的推理链路，不再"黑盒等待"。
- **US-032**：作为用户，我希望工具调用失败时 AI 能自动重试，从而减少因网络抖动导致的交互失败。
- **US-033**：作为用户，我希望编排 Agent 能自动拆解复杂任务并调用合适的专业 Agent 执行子任务，从而一站式解决复杂问题，无需手动切换 Agent。
- **US-034**：作为用户，我可以在设置中调整 ReAct 循环的最大迭代次数和工具重试次数，从而根据模型能力和任务复杂度灵活控制。
- **US-035**：作为用户，我希望 ReAct 过程展示不打断正常对话流程，存量对话不受影响，从而平滑升级。

## 范围

### 本次要做
- **FP-031**：ReAct 循环引擎 — 将当前硬编码的单次工具循环替换为可配置迭代次数的 ReAct 循环，支持 Thought→Action→Observation→Thought 的完整推理闭环。
- **FP-032**：推理过程流式展示 — 后端通过 SSE 流式推送 Thought/Action/Observation 事件，前端新增可视化组件区分展示"思考"、"工具调用"、"观察结果"和"最终回答"。
- **FP-033**：工具调用重试机制 — 工具执行失败时自动重试，重试次数可在设置中配置（默认 5 次），支持指数退避。
- **FP-034**：编排 Agent（Orchestrator）— 新增编排 Agent 类型，可拆解复杂任务为子任务，通过智能路由委派给 Worker Agent 执行，汇总结果后回复用户。
- **FP-035**：设置面板扩展 — 在设置中新增"ReAct 设置"区域，包含最大迭代次数（默认 5）、工具重试次数（默认 5）。
- **FP-036**：重试/停止控制 — 工具重试期间用户可点击"停止"中断重试；ReAct 循环期间用户可随时中断。

### 本次不做
- ReAct 循环中的 Plan-and-Solve 模式（先规划再执行），本次仅实现 ReAct（边推理边行动）。
- 多层级嵌套编排（编排 Agent 再调编排 Agent），本次仅支持单层 Orchestrator→Worker。
- Worker Agent 间的直接通信（A2A 协议），本次编排 Agent 作为唯一中枢。
- 工具调用结果的富展示（图表、图片等），保持文本回显。
- ReAct 过程的持久化存储（消息历史中仅保留最终回答和工具调用摘要，不保留完整 Thought 链）。
- 非流式的 ReAct 支持（标题生成等场景仍使用原有非流式逻辑）。
- 存量对话的历史消息迁移（仅新对话启用 ReAct 展示）。
- 自定义重试策略（如对不同工具设置不同重试次数），本次统一次数配置。

## 业务规则
- **BR-043**：ReAct 循环最大迭代次数从设置中读取，默认 5 次，取值范围 1~20。达到上限后 AI 必须输出最终回答，不再允许工具调用。
- **BR-044**：每次迭代中，AI 可选择调用 0~N 个工具（并行调用），所有工具执行完毕后进入下一轮迭代。
- **BR-045**：工具调用失败时自动重试，最大重试次数从设置中读取，默认 5 次，取值范围 0~10。重试间隔采用指数退避（1s、2s、4s、8s、16s）。
- **BR-046**：重试耗尽后仍失败时，将错误信息（截断至 500 字符）作为 Observation 传给 AI，由 AI 决定下一步（换工具或报错回退）。
- **BR-047**：SSE 事件增加新类型：`event: thought`（推理文本）、`event: action`（工具调用声明）、`event: observation`（工具结果）。前端根据事件类型分别渲染。
- **BR-048**：编排 Agent 的 system prompt 在基础 system prompt 之上追加"任务拆解与委派"指令。拆解后的子任务通过内部接口委派给对应的 Worker Agent。
- **BR-049**：编排 Agent 的 Worker Agent 列表继承当前可用 Agent 列表（含通用助手、天气查询、自定义 Agent），不可用 Agent 不参与委派。
- **BR-050**：Worker Agent 执行子任务时复用已有的 SSE 流式回复机制，但结果不直接推送给前端，而是由编排 Agent 汇总后统一输出。
- **BR-051**：ReAct 过程和编排 Agent 的子任务执行过程均遵循现有的暂停/停止机制（`req.on('close')` 中断流）。
- **BR-052**：编排 Agent 的总执行时间不超过 120 秒（含所有子任务），超时则输出已完成的子任务结果，未完成的返回超时提示。
- **BR-053**：在设置中修改 ReAct 迭代次数或重试次数后，下一次对话生效，不中断当前对话。

## 验收标准
- [ ] **AC-063**：AI 在单次回复中可连续调用最多 5 次工具（如搜索→写入文件→创建 Issue），链式执行成功，最终回答正确反映所有步骤结果。
- [ ] **AC-064**：前端实时展示 ReAct 过程——"思考"内容以斜体/不同颜色显示、"工具调用"以卡片形式显示（工具名+参数）、"观察结果"以折叠方式展示、"最终回答"正常渲染。
- [ ] **AC-065**：工具调用失败后自动重试，重试次数达到上限后 AI 收到错误信息并做出合理回应（如"抱歉，查询失败了"或"换一种方式试试"）。
- [ ] **AC-066**：在设置中将最大迭代次数改为 3，新对话中 AI 最多调用 3 次工具后必须输出最终回答。
- [ ] **AC-067**：在设置中将重试次数改为 0，工具调用失败立即返回错误给 AI，不再重试。
- [ ] **AC-068**：编排 Agent 收到复杂任务（如"查北京天气并分析是否适合跑步"）后，自动拆解为"查天气"和"分析"两个子任务，分别委派给对应 Agent，汇总结果后回复。
- [ ] **AC-069**：ReAct 循环或编排执行期间点击"停止生成"，AI 立即停止，已输出的内容保留。
- [ ] **AC-070**：编排 Agent 总执行时间超过 120 秒时，超时终止，已完成的子任务结果正常输出。
- [ ] **AC-071**：ReAct 过程中 SSE 连接异常断开时，已保存的消息完整（至少包含用户消息和已完成的最终回答），服务器无内存泄漏。
- [ ] **AC-072**：使用未绑定任何工具的 Agent（通用助手）进行普通对话，行为与 V1.5 完全一致，无 ReAct 相关干扰。
- [ ] **AC-073**：编排 Agent 的 Worker 列表中不包含不可用的 Agent（如天气 Agent 未配置环境变量时不可用）。

## 非功能性需求
- **NF-024**：ReAct 单次迭代中工具调用的响应时间不超过 30 秒（含 MCP Server 执行时间）。
- **NF-025**：编排 Agent 的总执行时间不超过 120 秒。
- **NF-026**：前端接收到 thought/action/observation 事件后，应在 100ms 内渲染到界面。
- **NF-027**：ReAct 循环的内存开销不超过每轮迭代 1MB（工具结果缓存），释放后 GC 及时回收。
- **NF-028**：重试指数退避最大等待时间不超过 16 秒。
- **NF-029**：ReAct 过程展示不显著增加页面 DOM 节点数，单次 ReAct 循环渲染节点不超过 200 个。

## 风险与依赖
- **依赖项**：
  - AI 模型需支持 Function Calling（已有，V1.2 已验证）。
  - AI 模型需支持 Thinking/Reasoning 能力（已有，V1.3 已验证）。
  - 编排 Agent 依赖路由服务的 Agent 列表接口（已有）。
- **风险项**：
  - 长 ReAct 链（>5 次迭代）可能导致 token 消耗急剧增加 → 应对：默认 5 次上限，用户可调低。
  - 工具调用结果过大（如读取大文件）可能导致 API token 超限 → 应对：对 Observation 内容截断（默认 2000 字符），配置化。
  - 编排 Agent 的子任务结果冲突或依赖（子任务 B 需要子任务 A 的结果）→ 应对：本次仅支持独立子任务的并行编排，不解决任务间依赖（Plan-and-Solve 模式延后）。
  - 重试死循环（工具持续失败持续重试）→ 应对：总次数上限 + 指数退避 + 用户随时可中断。
  - 前端兼容性：SSE 自定义事件类型（`event:` 字段）需浏览器支持 → 应对：所有主流浏览器均支持。
- **当前阻塞**：无

## 决策记录
- ReAct 实现位置：在 `aiProxy.ts` 中新增 `reactChat` 函数，与现有 `streamChat` 并存。`streamChat` 保留供非 ReAct 场景（标题生成等）使用。
- SSE 事件格式扩展：使用 `event:` 字段区分类型，而非在 data 中加 type 字段。前端 EventSource 或 ReadableStream 根据 event 类型分发渲染。
- 编排 Agent 类型：新增 `orchestrator` 类型，在 `agents` 表中通过 `type` 字段区分。不与 MCP 工具绑定，而是通过内置的"任务拆解→委派"能力工作。
- 重试逻辑位置：在 `toolRegistry.ts` 或新增 `retryWrapper.ts` 中实现统一的 RetryWrapper，而非在 `executeTool` 函数中硬编码。
- 编排 Agent 不加入自动路由，用户需手动选择（与当前 Agent 锁定机制一致），避免编排 Agent 被误触发导致意外行为。
