## 背景与目标
- **业务背景**：AI Chat V1.5 已具备完整的 Agent 体系——内置通用助手和天气查询、MCP 协议接入外部工具、自定义 Agent 绑定工具链、跨对话记忆。但 Agent 选择完全依赖用户手动切换，AI 无法自主判断应该使用哪个 Agent 或工具来响应用户请求。随着自定义 Agent 和 MCP 工具增多，手动切换的认知负担越来越重。
- **当前问题**：
  1. 用户每次提问都要先判断"这个问题该用哪个 Agent"，切换成本高，且容易选错。
  2. 复合型问题（如"北京明天冷吗，帮我安排一下出行"）需要用户自行拆解，无法一次完成。
  3. 用户不清楚每个 Agent 的能力边界，倾向于只用通用助手，导致 MCP 工具和自定义 Agent 使用率低。
  4. 无法为后续更复杂的"多步工具编排"场景积累路由数据和决策基础。
- **成功标准**：用户输入消息后，系统自动识别意图并路由到最合适的 Agent；路由错误时用户可手动纠正并反馈；路由决策对用户透明可理解；整体对话效率提升，Agent 切换成本归零。

## 用户与场景
- **目标用户**：所有 AI Chat 使用者（特别是配置了多个 Agent 和 MCP 工具的用户）
- **典型场景**：
  - 用户输入"北京明天天气怎么样"，系统自动路由到 Weather Agent，无需手动切换。
  - 用户输入"帮我读取桌面上的 todo.md 然后总结一下"，系统路由到文件助手 Agent，AI 调用 MCP 工具读取文件并总结。
  - 用户输入"你好，我叫小明"，系统识别为日常对话，路由到通用助手。
  - 用户手动切换到天气 Agent 后发送消息，下次发送新问题时系统重新根据内容自动路由。
  - 用户发现路由错误，手动切换到正确的 Agent 后继续对话。

## 用户故事
- **US-039**：作为用户，我希望系统自动理解我的问题意图并选择合适的 Agent 处理，从而无需手动切换。
- **US-040**：作为用户，我希望看到系统选择了哪个 Agent 以及为什么，从而了解路由决策的依据。
- **US-041**：作为用户，我仍然可以手动切换 Agent，手动选择后当前对话锁定为该 Agent，从而在需要时保持控制权。
- **US-042**：作为用户，当路由错误时，我可以通过手动切换来纠正，后续系统从中学习，从而逐步减少路由错误。
- **US-043**：作为用户，我希望路由决策在消息发送前完成，不增加首字到达延迟，从而不牺牲对话流畅性。
- **US-044**：作为用户，我希望未来的系统能自主规划多步骤作（查询天气 → 规划出行），从而一次提问完成复杂任务。

## 范围

### 本次要做
- **FP-039**：轻量意图识别 — 在消息发送前通过关键词匹配 + LLM 轻量分类（默认开启）双层识别用户意图，判断应路由到哪个 Agent。关键词匹配为主路径（<10ms），LLM 分类作为精度增强。
- **FP-040**：自动路由引擎 — 根据意图识别结果，自动选择最匹配的 Agent（通用助手、天气查询、自定义 Agent）。路由结果在 SSE 连接建立前确定。
- **FP-041**：路由透明度 — 在前端 ChatArea 中展示当前被自动选中的 Agent（如高亮或标签），让用户感知路由决策。
- **FP-042**：手动覆盖机制 — 用户可随时手动切换 Agent，手动选择后当前对话锁定为该 Agent，不再自动路由（解除锁定后可恢复）。设置中提供"路由模式"配置项（自动/手动），默认自动。
- **FP-043**：Agent 能力声明与内置关键词预设 — 每个 Agent 在注册时提供能力描述和 `triggerKeywords`。内置 Agent 预设关键词：通用助手（无关键词，作为兜底）、天气（天气、温度、预报、风力、降雨、晴、雨、雪、台风、湿度、空气质量）。自定义 Agent 在创建时填写。
- **FP-044**：工具编排预留扩展点 — 路由引擎设计时预留编排层接口（`router.beforeRoute`、`orchestrator.onRoutingComplete` 钩子），支持未来将单个复杂请求拆解为多步工具调用序列。
- **FP-045**：路由模式配置 — 在设置弹窗的 General 标签页中添加"路由模式"下拉选项（自动/手动），默认自动。自动模式下每次消息自动路由；手动模式下保持现有行为（用户手动选择）。
- **FP-046**：后端统一日志模块 — 建立结构化日志工具模块，规范全系统的日志输出格式和级别管理，为路由日志、调试追踪和运行监控提供统一的日志基础设施。

### 本次不做
- 多步工具编排（本次仅做单步路由决策，多步编排在下个版本引入，但当前路由引擎预留编排扩展点）。
- 基于历史路由反馈的自动学习（路由错误纠正数据暂不用于模型训练，仅记录日志）。
- Slot filling（多参数收集，如"从北京到上海"需拆出发地、目的地）。
- 路由 A/B 测试或路由质量评分面板。
- Agent 路由结果的用户反馈打分机制（如"这次路由正确吗"）。
- 路由决策树的可视化编辑界面。
- 跨对话路由偏好记忆（如用户总把"帮我写代码"路由到通用助手而非天气 Agent，暂不自动学习此偏好）。

## 业务规则
- **BR-054**：路由引擎在用户提交消息后、SSE 请求发起前执行，路由决策时间不超过 500ms。
- **BR-055**：路由判定依据包括：Agent 的 `description` 字段、`triggerKeywords` 字段和当前消息内容的关键词匹配，以及 LLM 轻量分类调用（默认启用）。
- **BR-056**：LLM 分类调用使用非流式模式，单次请求 token 消耗不超过 200 tokens（含 prompt 和 agent 列表）。关键词匹配置信度高（>0.8）时直接决策，不触发 LLM 调用以节省 token。
- **BR-057**：路由置信度低于阈值（默认 0.6）时，回退到通用助手，不做低质量路由。
- **BR-058**：用户手动切换 Agent 后，当前对话锁定为该 Agent，`conversations` 表记录 `lockedAgent` 字段，后续消息不再触发自动路由。
- **BR-059**：用户可通过点击"解锁"恢复自动路由，或新建对话自动恢复。
- **BR-060**：Agent 能力声明中的 `triggerKeywords` 支持正则表达式，支持按工具绑定的 MCP 工具描述自动推断关键词（工具已有 Function Calling Schema 中的 description 字段）。内置 Agent 关键词预设写死在代码中，自定义 Agent 关键词在创建时通过前端表单填写。
- **BR-061**：路由决策结果以结构化格式记录到 `routing_logs` 表中（含 `routeTo`、`confidence`、`method`、`latencyMs`、`timestamp`），用于后续分析和优化，不暴露给前端除 Agent 选择外的额外 UI。
- **BR-062**：后端统一日志模块要求：所有日志使用结构化 JSON 格式，包含 `timestamp`、`level`、`module`、`message`、`data` 字段。新建 `server/utils/logger.ts`（或 `server/services/loggerService.ts`）作为全局唯一的日志入口。路由日志通过日志模块按 `module: 'routing'` 输出，同时持久化到 `routing_logs` 表。

## 验收标准
- [ ] **AC-072**：用户发送"北京明天天气怎么样"，系统自动路由到 Weather Agent，SSE 请求中的 `agent` 参数为 `weather`。
- [ ] **AC-073**：用户发送"你好，帮我写一篇短文"，系统自动路由到通用助手，Agent 选择器显示选中"通用助手"。
- [ ] **AC-074**：用户发送与自定义 Agent 描述匹配的消息（如"读取我的日记文件"），系统自动路由到对应的自定义 Agent。
- [ ] **AC-075**：路由决策后，前端 Agent 选择器高亮显示被选中的 Agent，用户可感知路由结果。
- [ ] **AC-076**：用户手动切换 Agent 后，当前对话锁定，后续消息不再自动切换 Agent。
- [ ] **AC-077**：用户手动切换后点击"解锁"，当前对话恢复自动路由。
- [ ] **AC-078**：发送模糊消息（如"嗯"、"好的"），路由引擎回退到通用助手。
- [ ] **AC-079**：路由决策时间（从消息提交到路由确定）不超过 500ms，不影响首字到达时间。
- [ ] **AC-080**：新建对话首次消息触发自动路由，路由结果正确。
- [ ] **AC-081**：LLM 分类调用失败（API 超时/报错）时自动降级为关键词匹配，不阻断消息发送。
- [ ] **AC-082**：所有已有 Agent（通用助手、天气查询、自定义 Agent）均可被路由引擎识别和选择。
- [ ] **AC-083**：路由引擎预留的编排扩展点可通过配置启用，启用后不影响当前单步路由行为。

## 非功能性需求
- **NF-031**：路由决策 P99 延迟不超过 500ms（含 LLM 分类调用时的网络开销）。
- **NF-032**：LLM 分类调用每次最多消耗 200 tokens（请求 + 响应合计），prompt 中仅包含 Agent 名称和描述的精简列表。
- **NF-033**：路由引擎代码模块与 aiProxy 解耦，独立的 `routingService`，可独立测试。
- **NF-034**：编排扩展点接口（hook）在本次不启用情况下零性能开销（空函数模式）。
- **NF-035**：Agent 能力描述中的 `triggerKeywords` 支持热更新（修改 Agent 后路由引擎无需重启）。
- **NF-036**：统一日志模块支持分级日志输出（debug / info / warn / error），单条日志大小不超过 1KB，不影响主请求性能。

## 风险与依赖
- **依赖项**：
  - 现有 Agent 体系（`GET /api/agents` 返回的 `id`、`label`、`description` 字段）。
  - 现有 SSE 协议的 `agent` 字段传递机制。
  - 自定义 Agent 的 `description` 字段质量影响路由准确率。
- **风险项**：
  - LLM 分类调用增加一次额外 API 请求，可能增加消息发送延迟 → 应对：关键词匹配作为主路径（<10ms），关键词匹配高置信时跳过 LLM 调用。LLM 调用设 3s 超时，超时降级为关键词结果。
  - 路由错误会打断用户预期 → 应对：路由结果透明展示 + 手动覆盖机制兜底，确保路由错误可被快速纠正。
  - 用户对自动切换不适应 → 应对：设置中提供"路由模式"配置项（自动/手动），用户可按需切换。
  - 多个 Agent 间触发关键词重叠导致路由冲突 → 应对：定义关键词匹配优先级规则（精确匹配 > 正则匹配 > 描述匹配，匹配度高的优先）。
  - 路由日志依赖结构化日志基础设施，当前后端无统一日志模块 → 应对：本次新增统一日志工具模块，为全系统日志规范奠定基础。
- **当前阻塞**：无

## 已确认事项

以下为决策确认：

| 事项 | 决策 |
|------|------|
| 内置 Agent 预设 triggerKeywords | 通用助手无关键词（兜底）；天气预设：天气、温度、预报、风力、降雨、晴、雨、雪、台风、湿度、空气质量 |
| LLM 分类默认开启 | 默认开启。关键词高置信（>0.8）时直接决策跳过 LLM 调用以节省 token |
| 设置界面"路由模式"配置项 | 在设置 General 标签页中添加"路由模式"下拉选项（自动/手动），默认自动 |
| 后端统一日志模块 | 新建结构化日志工具模块，统一全系统日志输出 |

## 待确认事项
- 路由置信度阈值（0.6）是否合适？需上线后根据实际数据调整。
- 路由日志是否需要提供可视化面板（如路由历史查询）？本次仅记录日志和数据库存储，不提供面板。
- 是否需要对天气 Agent 做特殊处理——当关键词匹配到天气但环境变量未配置时，路由到通用助手并提示未配置？
- 统一日志模块的覆盖范围：仅本次路由系统使用，还是波及全系统（现有 server 中的 console.log / console.error 全部替换）？建议本次仅新功能使用，存量逐步迁移。
- 日志模块是否需要支持日志文件轮转（按大小/日期切分）？还是仅 stdout + DB 持久化？

## 附录：编排扩展点预留说明

### 设计意图
V1.6 聚焦"单步路由决策"，但路由引擎在设计时预留编排层接口，确保 V1.7 多步工具编排可平滑接入。

### 预留接口
| 接口 | 触发时机 | 返回值 | 用途（V1.7） |
|---|---|---|---|
| `beforeRoute(message, context)` | 路由决策前 | 可修改 `message` 或跳过路由 | 预处理器对消息做标准化、参数提取 |
| `onRoutingComplete(routeResult, context)` | 路由决策后、API 调用前 | 可修改 `routeResult` | 编排器根据路由结果注入额外上下文 |
| `shouldDecompose(message)` | 路由决策后 | `boolean` | 判断当前问题是否需要多步拆解 |
| `decomposeTask(message, agentId)` | `shouldDecompose` 返回 true | 子任务列表 | 将复杂问题拆解为有序的子任务序列 |

### 预留配置
```typescript
interface OrchestrationConfig {
  enabled: boolean;          // V1.6 固定为 false，V1.7 启用
  maxSteps: number;          // 最大编排步数（V1.7 默认 5）
  strategy: 'sequential' | 'parallel' | 'auto';  // 执行策略（V1.7 从 sequential 起步）
}
```

### 向前兼容保证
- V1.6 的路由引擎在 `enabled: false` 时行为与无扩展点时完全一致。
- V1.7 启用编排后，V1.6 的单步 Agent 配置和能力描述无需修改。
- 所有预留接口都有默认的空实现（noop），不产生性能开销。
