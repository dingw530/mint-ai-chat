## 背景与目标
- **业务背景**：AI Chat V1.1 已支持基础对话、系统提示词和思考模式。用户可以通过文本向 AI 提问，AI 仅依赖自身训练数据回答，无法获取实时外部信息。许多用户希望在对话中查询实时天气等信息，但当前 AI 模型的知识存在截止日期，且无法主动调用外部 API。
- **当前问题**：
  1. AI 无法获取实时数据（如天气），回答依赖训练数据中的过时信息，或明确拒绝回答。
  2. 需要手动复制信息到外部工具查询再回到对话使用，体验割裂。
- **成功标准**：用户询问天气时，AI 自动调用和风天气 API 获取实时预报数据，并以自然语言形式回复天气信息，无需用户手动切换工具。

## 用户与场景
- **目标用户**：所有 AI Chat 用户，尤其是需要查询实时天气的日常用户
- **典型场景**：
  - 用户输入"北京今天天气怎么样？"，AI 实时查询后回复北京当天天气情况。
  - 用户输入"上海下周会下雨吗？"，AI 查询 7 天预报后给出是否需要带伞的建议。
  - 用户输入"比较一下深圳和广州的气温"，AI 分别查询两个城市天气后给出对比。

## 用户故事
- **US-009**：作为用户，我希望在对话中直接询问某个城市的天气，AI 能给出实时准确的天气信息，从而无需切换应用或手动查询。
- **US-010**：作为用户，我希望 AI 能查询未来多天的天气预报（3 天或 7 天），从而可以提前规划出行或活动。
- **US-011**：作为用户，我希望天气信息查询对对话体验无侵入性——查询过程在后台自动完成，我不需要感知工具调用的存在。
- **US-012**：作为用户，我可以在前端界面选择切换到"天气查询"模式，从而明确告知 AI 当前需要查询天气，提高工具调用的准确率。
- **US-013**：作为用户，我可以在通用助手和天气查询 agent 之间自由切换，切换后对话上下文保持连续。

## 范围
### 本次要做
- **FP-009**：AI 工具调用能力（Server-side Function Calling）—— 后端 aiProxy 支持向 AI API 发送工具定义，接收并处理工具调用请求，执行外部 API 后返回结果给 AI 生成最终回复。
- **FP-010**：和风天气查询集成 —— 后端新增天气查询服务（城市名称解析 + 逐日天气预报），通过 JWT 认证访问和风天气 OpenAPI。
- **FP-011**：天气查询工具定义 —— 向 AI API 注册 `get_weather_forecast` 函数，描述语言为中文，参数包括城市名称和预报天数。
- **FP-012**：前端 Agent 选择器 —— 在 ChatArea 输入框上方新增 agent 选择器（"通用助手"和"天气查询"两个预设 agent），用户可点击切换当前使用的 agent 模式。

### 本次不做
- 前端展示天气信息的定制化 UI（如天气图标、温度条等）—— 保持纯文本回答
- 多轮工具调用（AI 在单次回复中连续调用多个工具）
- 非天气类工具的集成（预留工具注册机制，但本次仅实现天气查询）
- 天气查询的响应缓存（每次实时请求）
- 地理位置自动识别（需用户输入城市名称）
- 历史天气数据查询
- 空气质量、湿度、风速等扩展数据的展示（API 返回中不包含或由 AI 自行决定是否展示）

## 业务规则
- **BR-010**：工具调用（Function Calling）仅在后端 `aiProxy` 层实现，前端 SSE 协议不变，前端无感知。
- **BR-011**：天气查询需要 `QWEATHER_PROJECT_ID`、`QWEATHER_KEY_ID` 和 `QWEATHER_PRIVATE_KEY` 三个环境变量同时配置；任一缺失时，AI 请求中不发送工具定义，天气询问退化为普通对话（AI 使用训练数据回答）。
- **BR-012**：JWT Token 签发使用 EdDSA（Ed25519）算法，有效期 15 分钟，在 qweatherService 内部自动刷新，无需外部定时任务。
- **BR-013**：城市名称解析优先使用和风天气城市查询接口（v2/city/lookup），支持城市中文名（如"北京"）和经纬度坐标。
- **BR-014**：工具调用最多两轮：第一轮 AI 返回函数调用 → 后端执行 → 第二轮 AI 返回最终回复。防止递归工具调用。
- **BR-015**：工具调用过程中出现的任何异常（网络错误、API 返回错误、JSON 解析失败等），aiProxy 捕获异常并向客户端写入 SSE 错误消息，不影响后续对话。
- **BR-016**：前端 Agent 选择器默认选中"通用助手"。切换 agent 后，后续消息的请求体中携带 `agent` 字段标识当前 agent 类型。
- **BR-017**：切换 agent 不中断当前对话或清除消息历史，仅影响后续消息是否携带工具定义。
- **BR-018**："天气查询"agent 仅在 `QWEATHER_PROJECT_ID`、`QWEATHER_KEY_ID` 和 `QWEATHER_PRIVATE_KEY` 三个环境变量均已配置时才可用；环境变量缺失时该 agent 选项在 UI 中置灰或显示"不可用"提示。

## 验收标准
- [ ] **AC-015**：用户输入"北京今天天气怎么样？"，AI 回复中包含北京当天天气描述（温度、天气状况等），信息与和风天气 API 返回一致。
- [ ] **AC-016**：用户输入"上海未来三天的天气"，AI 回复中包含上海未来三天的预报信息。
- [ ] **AC-017**：未配置 `QWEATHER_PROJECT_ID`、`QWEATHER_KEY_ID` 和 `QWEATHER_PRIVATE_KEY` 时，询问天气问题 AI 正常回复（可提示无法获取实时天气或使用训练数据回答），不报错。
- [ ] **AC-018**：非天气类的普通对话（如"写一首诗"），工作流与 V1.1 完全一致，SSE 流式输出不受影响。
- [ ] **AC-019**：配置错误的 QWeather 凭证时，询问天气后 AI 返回友好的错误提示，不崩溃。
- [ ] **AC-020**：天气查询响应后，完整 AI 回复（含天气信息）正常持久化到 messages 表，刷新后可查看。
- [ ] **AC-021**：启用思考模式时查询天气，推理内容通过 SSE 正常推送到前端并展示，最终回复包含天气信息。
- [ ] **AC-022**：输入框上方展示 agent 选择器，默认选中"通用助手"，可点击切换至"天气查询"。
- [ ] **AC-023**：切换至"天气查询"后发送"北京天气"，AI 调用天气工具返回准确天气信息；切回"通用助手"后发送同样问题，AI 行为与工具启用前一致。
- [ ] **AC-024**：切换 agent 不导致当前会话中断、消息丢失或历史消息异常。
- [ ] **AC-025**：未配置 QWeather 环境变量时，"天气查询"agent 选项置灰不可选，并显示提示文字。

## 非功能性需求
- **NF-008**：天气查询引入的额外延迟不超过 5 秒（正常网络条件下，含城市查询 + 预报查询 + AI 二次调用）。
- **NF-009**：新增 `jose` npm 依赖，JWT 使用 EdDSA（Ed25519）签名。
- **NF-010**：QWeather API 凭证在服务端配置，不暴露到前端。

## 风险与依赖
- **依赖项**：
  - AI 模型需支持 Function Calling / Tool Use 能力（GPT-4o、Claude 3.5+ 等均支持）。
  - 和风天气 API 服务可用性及账号有效性。
  - 环境变量 `QWEATHER_PROJECT_ID`、`QWEATHER_KEY_ID` 和 `QWEATHER_PRIVATE_KEY` 需由服务部署方从和风天气控制台获取。
- **风险项**：
  - 部分 AI 模型可能不支持 Function Calling，导致工具定义被忽略，天气询问退化为普通回答。
  - 和风天气免费套餐可能有调用频率限制（如 1000 次/天），超限后查询失败。
  - 工具调用引入的额外延迟可能影响用户体验（首字到达时间增加）。
- **应对建议**：
  - 工具定义始终作为可选增强，不破坏基础对话能力。
  - API 调用失败时返回友好错误提示，引导用户稍后重试。
  - 第二次 AI 调用（含工具结果的最终回复）采用流式输出，减少等待感知。

## 待确认事项
- 和风天气 API 免费套餐的具体调用限制和并发限制。
- 是否需要对同一城市的连续查询做短时间缓存（如 5 分钟内同一城市直接返回上次结果）？
- 是否支持城市名简写（如"沪"对应上海）？当前通过和风天气 API 处理。
