# 产品规格：图片生成模型支持

> 精简模式说明：功能点 3 个、路径单一、单系统单团队，符合精简模式条件。
> 省略非功能性需求、风险与依赖章节；用户故事采用简洁描述。

## 背景与目标

- **业务背景**：Mint · 清言 目前仅支持文本对话模型（Chat Completions / Messages API），用户无法使用图像生成模型（如 `gpt-image-2`）。
- **当前问题**：用户的 API 中转站已支持 `gpt-image-2` 等图像生成模型，但项目缺乏模型分类机制和独立的图片生成入口，无法利用图像生成能力。
- **成功标准**：用户能通过独立的图片生成入口，选择已配置的图片模型、输入 prompt 和参数，成功生成并查看图片。

## 用户与场景

- **目标用户**：使用中转站 API 且拥有图像生成模型权限的用户
- **典型场景**：用户在对话外需要生成图片时，切换到图片生成入口，选择图片模型（如 `gpt-image-2`），输入 prompt 并设置参数，生成并查看图片

## 用户故事

- **US-001**：作为用户，我希望在配置模型端点时能标记分类（文本/图片），从而让系统区分对话模型和图片模型。
- **US-002**：作为用户，我希望通过独立的图片生成入口进入生图界面，从而在纯文本对话外使用图片能力。
- **US-003**：作为用户，我希望能输入 prompt 并设置图片参数（尺寸、质量、格式），从而按需生成图片。
- **US-004**：作为用户，我希望生成的图片能显示在界面中，从而查看和使用图片。

## 范围

### 本次要做

- **FP-001**：模型端点分类 — `model_endpoints` 表新增 `category` 字段（`text`/`image`），端点点编辑/创建表单增加分类下拉选择
- **FP-002**：图片生成入口 — 在侧边栏或主导航增加「图片生成」独立入口，进入后展示图片生成界面（prompt 输入、参数配置、生成按钮、结果展示）
- **FP-003**：图片生成服务端 API — 新增 `POST /api/images/generate` 接口，调用 Images API 非流式生成图片，返回图片 URL

### 本次不做

- 图生图/图片编辑（`/v1/images/edits`），留待后续
- 图片生成历史记录持久化到数据库，首版仅运行时展示
- Python 服务端支持
- 图片管理（收藏、下载到本地等）

## 业务规则

- **BR-001**：标记为 `image` 类别的端点在聊天 ModelSwitcher 中不显示，仅在图片生成入口中可选
- **BR-002**：标记为 `text` 类别的端点在图片生成入口的模型选择器中不显示
- **BR-003**：图片生成调用 `POST /v1/images/generations`，请求为非流式（非 SSE），直接返回 JSON 结果
- **BR-004**：图片生成请求参数包括 `model`、`prompt`、`size`、`quality`、`output_format`、`response_format`、`n`（固定为 1）
- **BR-005**：至少需要配置一个 `image` 类别端点才能进入图片生成模式；无可用图片端点时显示引导配置提示
- **BR-006**：生成成功后图片 URL 保存在前端状态中；切换页面或刷新后丢失（首版不持久化）

## 验收标准

- [ ] **AC-001**：端点点编辑表单中可选择分类（文本/图片），保存后 `category` 字段正确写入数据库
- [ ] **AC-002**：分类为「图片」的端点在聊天标题的 ModelSwitcher 下拉菜单中不出现
- [ ] **AC-003**：分类为「文本」的端点在图片生成入口的模型选择器中不出现，仅有图片端点可选
- [ ] **AC-004**：侧边栏存在「图片生成」入口按钮，点击后切换到图片生成界面
- [ ] **AC-005**：图片生成界面包含 prompt 输入框、尺寸选择、质量选择、格式选择、生成按钮
- [ ] **AC-006**：点击生成后显示加载态，完成后图片展示在结果区域
- [ ] **AC-007**：生成失败时显示错误信息（如 API 密钥无效、模型不支持等）
- [ ] **AC-008**：无可用图片端点时，显示引导提示"请先在设置中配置一个图片模型端点"
- [ ] **AC-009**：`POST /api/images/generate` 接口返回 `{ created, data: [{ url, revised_prompt }] }` 格式

## 待确认事项

- `size` 参数的可选项列表是否硬编码在前端，还是由服务端下发？首批暂定硬编码常用尺寸：`1024x1024`、`1536x1024`、`1024x1536`、`2048x2048`、`3840x2160`
- `quality` 的可选项：`low`、`medium`、`high`、`auto`，是否所有图片模型都支持四档？暂定前端硬编码，选 `auto` 作为默认值
- `output_format` 的可选项：`png`、`jpeg`、`webp`，是否满足首批需求？
