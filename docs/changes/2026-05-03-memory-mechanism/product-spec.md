## 背景与目标
- **业务背景**：AI Chat V1.4 已具备完整的对话管理、流式 AI 交互、MCP 工具集成和自定义 Agent 能力。但每次对话 AI 都是从零开始，无法记住用户在前序对话中分享的个人信息、偏好和反馈。用户每次对话都需要重复提供上下文，体验割裂。
- **当前问题**：
  1. AI 对用户没有"记忆"——换一个对话就要重新介绍自己、重新说明偏好。
  2. 用户反馈（如"回答太长了"、"用中文回答"）只在当前对话生效，无法跨对话持续改进 AI 行为。
  3. 项目相关的技术栈、业务背景等信息无法沉淀，每次对话需要重复告知。
  4. 没有可视化的记忆管理手段，用户无法知道 AI 记住了什么，也无法纠正错误记忆。
- **成功标准**：AI 能自动从对话中提取关键事实并跨对话记住；用户可在设置中查看/编辑/删除记忆；开启记忆后 AI 回答质量因上下文连贯性提升；用户可随时关闭记忆功能。

## 用户与场景
- **目标用户**：所有 AI Chat 使用者
- **典型场景**：
  - 用户在第一轮对话中说"我叫张三，是一名前端工程师"。后续对话中 AI 直接称呼"张三"并了解其前端背景。
  - 用户说"回答请尽量简洁"。后续所有对话中 AI 自动控制回答篇幅，不再冗长。
  - 用户说"我正在用 React + TypeScript 开发一个后台管理系统"。后续对话中 AI 给出的建议和技术方案自动匹配该技术栈。
  - 用户打开记忆面板，看到 AI 自动提取的十几条记忆，删掉一条不准确的，修改一条分类错误的。

## 用户故事
- **US-029**：作为用户，我希望 AI 在对话完成后自动提取关键信息（如我的名字、职业、偏好）并保存为记忆，从而无需手动记录。
- **US-030**：作为用户，我希望记忆按分类整理（个人信息、偏好、反馈、项目等），从而快速找到关注的信息。
- **US-031**：作为用户，我可以在设置中查看所有记忆列表，从而了解 AI 记住了关于我的哪些信息。
- **US-032**：作为用户，我可以编辑或删除单条记忆，从而纠正错误或过时的信息。
- **US-033**：作为用户，我可以手动添加一条记忆，从而让 AI 记住我想让它知道的信息。
- **US-034**：作为用户，我可以按分类筛选记忆列表，从而快速定位特定类型的记忆。
- **US-035**：作为用户，我可以在设置中一键开启或关闭记忆功能，从而按需控制 AI 是否参考历史记忆。
- **US-036**：作为用户，当记忆功能开启时，AI 在回答中会参考已有的记忆，从而获得更个性化的体验。

## 范围

### 本次要做
- **FP-030**：记忆自动提取 — 每次 AI 回复完成后，异步调用 AI 从该轮对话中提取关键事实，按分类格式化输出。
- **FP-031**：记忆分类体系 — 提取时自动分类（personal / preference / feedback / project / goal / general），注入时按分类分组展示给 AI。
- **FP-032**：记忆持久化 — 提取的记忆存入 SQLite `memories` 表，支持精确去重。
- **FP-033**：记忆注入 — 开启记忆时，已有记忆作为 system message 注入到每次对话的上下文中。
- **FP-034**：记忆管理面板 — 在设置弹窗中新增"记忆"标签页，支持查看、按分类筛选、编辑、删除、手动添加记忆。
- **FP-035**：记忆开关 — 在设置中添加记忆功能的启用/停用开关。
- **FP-036**：记忆 CRUD API — 后端 RESTful 接口支持记忆的增删改查和分类筛选。

### 本次不做
- 基于向量嵌入的语义检索（仅做精确匹配去重，不做模糊相似度匹配）。
- 记忆重要性评分和自动淘汰机制（所有记忆等同对待）。
- 按对话范围隔离记忆（所有对话共享同一记忆池）。
- 记忆之间的关联推理（如"用户提到 React" + "用户提到前端" → 自动关联）。
- 记忆同步到外部存储或云端备份。
- 记忆的导入/导出功能。
- Python 后端的记忆支持（仅 TS 后端）。
- 对话中选择性注入（目前是所有记忆全部注入或全部不注入）。

## 业务规则
- **BR-041**：记忆提取在 AI 回复保存完成后异步触发，不阻塞用户界面，提取失败不影响聊天体验。
- **BR-042**：提取时使用与主聊天相同的 AI API 端点/Key/模型，非流式调用，使用专用 extraction prompt。
- **BR-043**：提取结果格式为 `[category] content`，每行一条。无法解析格式的行丢弃不存。
- **BR-044**：新提取的记忆与已有记忆做精确内容匹配（`content` 字段完全一致），匹配到则不再入库，避免重复。
- **BR-045**：记忆注入时，按 personal → preference → feedback → project → goal → general 的顺序分组，每组前带 emoji 图标。
- **BR-046**：记忆注入位置在用户自定义 system prompt 之后、对话历史之前，作为独立的 system message。
- **BR-047**：记忆功能默认关闭，用户需在设置中手动开启。
- **BR-048**：手动添加的记忆来源标记为 null（无 source_conversation_id）；自动提取的记忆记录来源对话 ID。
- **BR-049**：删除记忆为物理删除，不可恢复。

## 验收标准
- [ ] **AC-058**：开启记忆后，完成一轮对话，在记忆面板中可看到 AI 自动提取的记忆，分类正确。
- [ ] **AC-059**：再次发起新对话，发送与记忆相关的提问，AI 的回复参考了已有记忆中的信息。
- [ ] **AC-060**：在记忆面板中按分类筛选，只显示对应分类的记忆。
- [ ] **AC-061**：在记忆面板中编辑一条记忆的内容和分类，保存后内容更新，注入时使用新内容。
- [ ] **AC-062**：在记忆面板中删除一条记忆，刷新后该记忆不再出现，注入时也不再包含。
- [ ] **AC-063**：在记忆面板中手动添加一条记忆（内容 + 分类），保存后生效，下轮对话中 AI 可参考。
- [ ] **AC-064**：关闭记忆开关后，记忆不再注入到对话中，已有记忆保留不删除。
- [ ] **AC-065**：重新开启记忆开关，记忆恢复注入。
- [ ] **AC-066**：连续多次提取到相同事实（如用户重复自我介绍），记忆表中不产生重复记录。
- [ ] **AC-067**：提取过程不影响主对话体验——流式回复不卡顿、SSE 不中断、不增加用户等待时间。

## 非功能性需求
- **NF-024**：记忆提取 API 调用超时限制为 10 秒，超时静默放弃本次提取。
- **NF-025**：单条记忆内容长度不超过 500 字符。
- **NF-026**：记忆总数不超过 200 条（超过时不再自动提取新记忆，需用户手动清理）。
- **NF-027**：记忆注入的 system message 总长度不超过 3000 tokens（借助后续评估调整）。
- **NF-028**：提取操作的内存消耗控制在 50MB 以内。

## 风险与依赖
- **依赖项**：
  - AI API 需支持非流式聊天补全（已有 `generateTitle` 复用该能力）。
  - 提取效果依赖于 AI 模型的理解和分类能力。
- **风险项**：
  - AI 可能提取到不准确或错误的信息 → 应对：提供记忆编辑/删除能力，prompt 强调"只提取确定的"。
  - 记忆过多导致 context window 被占用 → 应对：设定 200 条上限，监控注入 token 量，后续可优化为选择性注入。
  - ~~提取增加 API 调用次数和 token 消耗 → 应对：较为轻量的非流式调用 + 异步触发不阻塞主流程。~~（v1.5.1 通过价值判断机制大幅减少无效提取）
- **当前阻塞**：无

## v1.5.1 优化：价值驱动的记忆提取

### 背景
v1.5.0 中每次 AI 回复后都触发记忆提取，导致大量无效 API 调用（用户发"你好"、"帮我写代码"等无个人信息的内容也会触发提取）。实际运行中 >70% 的对话轮次不包含值得记忆的信息，造成 token 浪费。

### 优化目标
提取前增加低成本的价值判断环节，只有判定当前对话包含值得记忆的信息时，才调用 LLM 提取 API。

### 用户故事
- **US-037**：作为用户，我希望系统只在对话包含值得记忆的信息时才调用提取 API，从而减少不必要的 token 消耗和服务延迟。
- **US-038**：作为用户，我希望价值判断足够灵敏，不会遗漏真正重要的信息（如个人信息、偏好、反馈），从而保持记忆的完整性。

### 范围
#### 本次要做
- **FP-037**：价值判断机制 — 在提取前增加启发式规则预检查，快速过滤明显无价值的对话轮次。
- **FP-038**：可配置阈值 — 价值判断的灵敏度可在 settings 中配置（可选，后续迭代）。

#### 本次不做
- 基于 LLM 的价值预判断（用 AI 判断是否需要提取 → 成本反而增加，失去优化意义）。
- 用户自定义价值判断关键词（v1 用固定规则，后续可按需扩展）。
- 提取频率统计面板（API 调用次数可视化）。

### 业务规则
- **BR-050**：提取触发前必须先经过价值判断，判断不通过则跳过本次提取，不调用 AI API。
- **BR-051**：价值判断使用启发式规则（见设计文档），在服务端同步执行，不产生额外 API 调用。
- **BR-052**：价值判断通过的标准是——用户消息内容包含至少一个"自指模式"（自我介绍、偏好声明、反馈、项目信息等个人相关内容）。
- **BR-053**：判断不通过时，只跳过本次提取，不影响后续轮次的判断。

### 验收标准
- [ ] **AC-068**：发送无个人信息的消息（如"你好"、"今天天气怎么样"），不触发提取 API 调用。
- [ ] **AC-069**：发送自我介绍（如"我叫张三"），触发提取 API 调用并成功沉淀记忆。
- [ ] **AC-070**：连续的无关对话（>5 轮），不产生任何提取 API 调用。
- [ ] **AC-071**：价值判断不通过的轮次，不阻塞后续正常轮次的提取。

### 非功能性需求
- **NF-029**：价值判断的执行时间不超过 5ms（纯正则/字符串匹配，不涉及网络调用）。
- **NF-030**：价值判断的误判率（遗漏真正有价值的信息）不超过 10%。

## 决策记录
- 记忆存储位置：新建 `memories` 表，独立于 `messages` 和 `settings`。
- 提取方式：复用 AI API 非流式调用（与 `generateTitle` 相同机制），不引入额外依赖。
- 去重策略：精确匹配 `content` 字段（V1），不引入向量数据库或模糊匹配。
- 注入策略：全部记忆按分类分组注入（V1），后续可优化为相关性排序。
- 分类体系：6 个预定义分类（personal / preference / feedback / project / goal / general），不支持用户自定义分类。
- 记忆开关：settings key-value 存储（`memoryEnabled: 'true'/'false'`），与 `thinkingMode` 模式一致。
- ~~提取触发时机：每次 AI 回复持久化完成后立即触发，而非对话结束时。~~（已废弃，见 v1.5.1）
- v1.5.1 优化：提取前增加价值判断环节，仅在有价值时触发 LLM 提取，减少无效 API 调用。
