# sdd-doc

开发流程始于需求理解与澄清，随后生成三类核心文档：`product-spec`（产品规格/PRD）、`design-doc`（设计方案）、`exec-plan`（执行计划）。

## 命令参考

```
/sdd-doc-generator [spec|design|plan|apply|verify|check-doc|archive|pipeline] [业务主题] [+补充约束]"
```

**参数规则**：第一个参数匹配文档类型（支持别名如 `prd`→`spec`、`方案`→`design`、`计划`→`plan`、`实现`→`apply`），后续合并为主题，`+`开头为补充约束。

| 命令 | 说明 |
|------|------|
| `discuss 主题` | 需求理解与澄清：复述需求理解、主动提出澄清问题、与用户对齐后进入下一步 |
| `spec 主题` | 理解需求并生成产品规格（用户故事、验收标准），需求模糊时主动澄清 |
| `design 主题` | 生成设计方案（多方案对比、接口定义） |
| `plan 主题` | 生成执行计划（任务拆解、验收方式） |
| `apply 主题` | 开发实现：据文档编码，逐 TP 推进 |
| `check-doc 主题` | 文档评审：检查完整性、追溯链路、质量 |
| `verify 主题` | 实现审计：隔离 agent 检查代码与文档一致性 |
| `archive 主题` | 归档已完成变更 |
| `pipeline 阶段1→阶段2 主题` | 串联多个阶段顺序执行，共享上下文 |
| `changes` | 列出所有活跃变更 |
| _(无参数)_ | 交互式引导 |

**自然语言触发**：无参数时按关键词推断类型——讨论/理解/澄清/需求分析→discuss，需求/验收→spec，技术/方案→design，步骤/排期→plan，编码/实现→apply，检查/追溯→check，审计/审核→verify。

### check-doc — 文档评审

对变更进行文档完整性、追溯链路与质量评估，输出评审报告与评级，不修改文档。

**评审维度**：

| 维度 | 检查内容 | 等级 |
|------|---------|------|
| 结构完整性 | 文件齐全、模板完整、精简模式注明原因 | 必检 |
| 追溯链路 | ID 引用有效、无孤立项、traceability 同步 | 必检 |
| 信息质量 | 填充率、"待确认"残留、验收标准可验证 | 必检 |
| 方案决策 | design-doc 有 ≥2 方案、对比表、明确决策 | 选检 |

**评级**（短板原则）：A=所有维度通过，B=主要通过少量改善，C=多维度不达标，D=关键缺失。

**流程**：扫描目录→解析 ID 验证追溯→统计填充率→短板定级→输出报告。

### verify — 实现审计

对代码实现进行全面审计。**必须启动隔离 agent**，避免上下文惯性遗漏。审计拆分为两个相互隔离的 agent，各自独立执行后汇总报告。

| 检查项 | 严重级别 | 执行者 |
|--------|---------|--------|
| 功能覆盖：spec 中 US/FP/AC 是否实现 | ❌ 错误 | consistency-auditor |
| 设计一致性：DS/API 设计与代码一致 | ❌ 错误 | consistency-auditor |
| scope 偏差：代码与文档互有遗漏 | ⚠️ 警告 | consistency-auditor |
| 记录完整性：exec-plan vs 代码产出匹配 | ⚠️ 警告 | consistency-auditor |
| 代码规范合规：扫描项目约定文件，分析代码规范约束，检查代码是否合规 | ⚠️ 警告 | convention-auditor |

**流程**：

1. **启动 consistency-auditor（文档一致性审计）**：隔离 agent → 读取变更文档 → git diff 定位代码 → 正向追溯（文档→代码逐项检查） → 反向追溯（代码→文档查遗漏） → 输出审计报告
2. **启动 convention-auditor（代码规范合规检查）**：隔离 agent → 扫描项目约定文件 → 分析代码规范约束 → git diff 定位代码 → 逐条检查合规性 → 输出审计报告
3. **汇总报告**：合并两份审计报告，短板定级 → 偏差修复后重新审计直至一致

consistency-auditor 与 convention-auditor 彼此隔离，互不可见，确保各自判断不受对方影响。

**使用示例**：
| 说法 | 效果 |
|------|------|
| `我们先讨论一下这个需求` | 需求理解与澄清，复述理解、提出疑问 |
| `写个 PRD：用户权限回收流程` | 生成 product-spec |
| `出个方案：审批弹窗解耦改造` | 生成 design-doc |
| `拆解任务：账号迁移需求` | 生成 exec-plan |
| `开始开发主账号注销前校验` | 进入编码模式 |
| `帮我检查文档完整度` | 文档评审 |
| `审计代码是否按文档实现` | 实现审计 |

### pipeline — 流程串联

按顺序自动执行多个阶段，前一阶段的输出（如变更目录）自动传递给后续阶段。

**语法**：
```
/sdd-doc-generator pipeline 阶段1→阶段2→阶段3 业务主题 [+补充约束]"
```

阶段间用 `→`（或 `->`）分隔。当前支持串联的阶段：`spec`、`design`、`plan`、`apply`、`verify`、`check-doc`、`archive`。

**常用管道**：

| 管道 | 说明 |
|------|------|
| `spec→design→plan` | 完整文档生成 |
| `plan→apply→verify→archive` | 开发实施到归档 |
| `spec→design→plan→apply→verify→archive` | 全生命周期 |

**流程**：
1. 解析管道阶段序列，校验每个阶段是否有效
2. 首个写入变更的阶段创建目录（`docs/changes/YYYY-MM-DD-业务主题/`），后续阶段共享该目录
3. 依次执行每个阶段，前一阶段失败则终止管道并提示
4. 所有阶段完成后更新 `traceability.md`
5. 输出各阶段执行摘要

**约束传递**：可在主题后附加全局约束（`+约束`），作用于所有阶段。

**使用示例**：

| 说法 | 效果 |
|------|------|
| `pipeline spec→design→plan 用户权限回收` | 生成完整三件套文档 |
| `pipeline plan→apply→verify→archive 审批弹窗改造` | 从计划到归档全流程 |
| `pipeline spec→design→plan→apply→verify→archive 账号迁移 +精简模式` | 全生命周期（精简模式） |

## 追溯链约定（Traceability）

| 前缀 | 文档 | 含义 |
|------|------|------|
| US-/FP-/BR-/AC-/NF- | product-spec | 用户故事/功能点/业务规则/验收标准/非功能需求 |
| DS-/API- | design-doc | 设计决策/接口契约 |
| TP- | exec-plan | 执行任务 |

**追溯方向**：
```
product-spec            design-doc              exec-plan
US-001 ──────────────> DS-001 ──────────────> TP-001
FP-001 ────> DS-002 ──────────────> TP-002
AC-001 ──────────────────────────────> TP-003
```

design-doc 需在需求追溯表中注明覆盖情况（完全/部分/下一版），exec-plan 在追溯总览汇总全链路。文档更新时新增条目接续最大编号，不重排已有编号。

## 文档方法论

### product-spec — 定义"做什么"
**适用场景**：需求澄清、范围定义、验收对齐
**核心问题**：这个产品要解决什么？为谁解决？怎么验证？
**必含章节**：背景与目标、用户与场景、用户故事、范围（含非目标）、业务规则、验收标准、风险与依赖
**要求**：语言清晰面向业务与协作方；验收标准可测试；未知写"待确认"禁止脑补。
**精简模式**：可省略非功能性需求、风险与依赖章节；用户故事用 1-2 句简洁描述。

### design-doc — 定义"怎么做"
**适用场景**：技术方案设计、架构调整、关键决策评审
**核心问题**：有哪些可行方案？优劣？选哪个？为什么？
**必含章节**：背景与目标、约束与前提、方案选项（≥2）、方案对比、最终决策、详细设计、影响与风险、发布与验证
**要求**：强调方案取舍；对现有系统改动须写清兼容性、发布与回滚。
**精简模式**：可省略方案选项/对比/最终决策章节，直接从约束跳到详细设计，在背景中说明方案唯一原因。

### exec-plan — 定义"如何落地"
**适用场景**：开发排期、任务拆解、执行协同
**核心问题**：分哪些阶段？每阶段做什么？谁能验收？怎么跟踪？
**必含章节**：目标与完成定义、背景与范围、前置条件、分阶段步骤、风险与依赖、验证与验收
**要求**：步骤可执行可跟踪可验收；优先按阶段组织，避免松散 todo 列表。
**精简模式**：可省略前置条件、风险与依赖；任务平铺代替分阶段结构。

## 精简模式

同时满足以下条件时启用：1-3 个功能点、路径单一、无跨团队依赖。裁剪规则：

| 文档 | 可省略章节 |
|------|-----------|
| product-spec | 非功能性需求、风险与依赖；用户故事简写 |
| design-doc | 方案选项/对比/最终决策，直接从约束跳到详细设计 |
| exec-plan | 前置条件、风险与依赖；任务平铺代替分阶段 |

精简模式下仍需分配追溯 ID，核心章节保留，并在背景中注明原因。信息量超预期时自动升级为完整模式。

## 需求理解与澄清

需求理解是**整个流程的前置必选步骤**，在开始任何文档撰写或编码之前执行。

### 执行要点

1. **阅读原始需求**：仔细读取用户提供的原始需求文档、PRD、纪要、邮件等所有相关材料
2. **复述理解**：用自己的话向用户复述对需求的理解，包括：
   - 需求的背景与目标——"为什么要做这个？"
   - 核心要解决的问题——"具体解决什么？"
   - 预期的产出和验收方式——"怎么做算完成？"
3. **主动澄清**：识别需求中的模糊、矛盾或缺失信息，主动提出澄清问题，示例：
   - "这里的XX是指什么？是否有更详细的定义？"
   - "关于XX场景，是否有更具体的说明？"
   - "XX和XX之间的关系是什么？是否有冲突？"
   - "这个需求的优先级/紧急程度如何？"
   - "有没有参考实现或类似案例？"
4. **达成共识**：与用户确认需求理解一致后再进入后续步骤
5. **记录待确认项**：仍有未解答的问题用"待确认"标记，禁止脑补

### discuss 命令

`/sdd-doc-generator discuss 主题` — 仅为需求理解与澄清，不生成文档。适合需求尚不明确、需要先对齐理解的场景。

### spec 中的需求理解

`spec` 阶段自动包含需求理解步骤，不会跳过澄清直接撰写。pipeline 中 `spec` 作为首阶段时同样先执行需求理解。

## 生成流程

### 步骤0：需求理解与澄清（必选前置）
执行[需求理解与澄清](#需求理解与澄清)中的全部要点。完成后才进入后续步骤。

### 步骤1：提炼输入
收集文档主题、业务背景/当前问题、目标用户或使用方、目标与非目标、已知约束、风险与待确认项。

### 步骤2：评估文档规模
| 判断维度 | 完整模式 | 精简模式 |
|---------|---------|---------|
| 功能点数量 | ≥ 4 个或不确定 | 1-3 个 |
| 改动路径 | 存在多个可行方案 | 路径单一、方案明确 |
| 协作范围 | 跨团队或多系统 | 单团队、单系统 |
| 文档受众 | 需多方评审对齐 | 主要用于内部记录 |

### 步骤3：补齐上下文
- 将模糊目标改写为可执行表述，零散需求归并为功能点/步骤/决策项
- 用"待确认"标记未知信息
- **禁止**编造接口、字段、时间、负责人、未经确认的结论、无来源的实现细节

### 步骤4：追溯编号
按顺序分配 ID（US-001, FP-001, DS-001, TP-001...）。检查 `docs/changes/` 下同主题目录或 `docs/product-specs/` 等目录下同名文件，继承最大编号。若生成 design-doc 或 exec-plan，自动关联对应 spec/design 建立追溯表。

### 步骤5：选择模板填充
模板位于 `templates/` 目录下。按步骤2判定的模式选择完整或精简模板，填充时自动关联上下级追溯 ID。

### 步骤6：生成 traceability.md
当具备 product-spec、design-doc、exec-plan 中至少两份文档时，自动生成追溯总览，含全链路追溯表和快捷链接。

## 落库目录

```
docs/changes/YYYY-MM-DD-业务主题/
├── product-spec.md
├── design-doc.md
├── exec-plan.md
├── traceability.md
└── spec-increment.md
```

快捷索引目录（`docs/product-specs/`、`docs/design-docs/`、`docs/exec-plans/`）保留为 README 索引视图。

## 输出规范

- 中文、纯 Markdown、保留模板章节、信息不足写"待确认"
- 优先输出成品文档，不额外输出大段解释

## 文档更新反馈

基于上一版更新时，正文后追加变更摘要与验证点：

```markdown
## 本次更新摘要
- 对比基线：{上一版标识}
- 新增/修改/删除：{以章节粒度归纳}
- 仍待确认：{...}

## 验证点
- 受影响验收标准：{...}
- 建议测试/回归点：{...}
- 发布观察点（可选）：{...}
```

## 执行与编码

用户发出"开始开发/实现需求/编码实现"等指令时进入编码模式。

### 执行前准备
1. 定位当前变更的 exec-plan，traceability.md 状态改为**进行中**
2. 初始化所有 TP 状态为"待启动"

### TP 实现流程（每 TP 四步）
1. **更新进度为进行中**：开始 TP 前，将 exec-plan 执行记录中该 TP 状态改为"进行中"，同步更新 traceability.md 追溯总览表中对应行状态为"进行中"
2. **理解上下文**：TP 描述、关联 DS（设计决策）、关联 AC（验收标准）
3. **代码实现**：严格遵循项目架构（分层依赖、V6 接口规范）；参照现有模式；新增方法加 JSDOC；枚举值引用常量文件
4. **自我验证**：对照 AC/DS 验证；检查边界条件与异常处理；模板无复杂逻辑
5. **更新进度为已完成**：TP 完成后，将 exec-plan 执行记录中该 TP 状态改为"已完成"，记录产出文件、问题与方案、偏差；同步更新 traceability.md 追溯总览表中对应行状态为"已完成"

### 执行中同步
- **开始每个 TP 前**：将 exec-plan 执行记录中该 TP 状态改为"进行中"、traceability.md 对应行改为"进行中"
- **完成每个 TP 后**：将 exec-plan 执行记录中该 TP 状态改为"已完成"、traceability.md 对应行改为"已完成"
- 发现 scope 变更在 plan 追加新 TP 并标注
- 执行记录**只追加不覆盖**

### 偏差处理
| 偏差类型 | 处理 |
|---------|------|
| 设计不合理/无法实现 | 暂停，与用户讨论调整 design-doc |
| 遗漏需求/scope变更 | 追加 TP，更新 product-spec |
| 技术约束不符 | 调整方案，同步更新 design-doc |
| 优先级变化 | 调整 TP 顺序/状态 |

### 完成归档
1. 执行 `verify` 实现审计（隔离 agent 检查功能覆盖、设计一致性、scope 偏差、记录完整性）
2. 审计通过后：traceability.md 改为**已完成**并填写完成日期 → 同步索引 → 存档审计报告
3. 审计不通过则修复后重新审计

### Handoff
执行记录中写明当前进度、下一步、阻塞/风险、关键决策。接手 agent 第一件事读取执行记录。

## 核心原则

1. **未知信息写"待确认"，禁止脑补**
2. `product-spec` 必须保留：用户故事、范围、验收标准
3. `design-doc` 必须保留：方案对比、最终决策、兼容与发布策略
4. `exec-plan` 必须保留：阶段目标、具体任务、验收方式
5. **执行阶段务必反向更新 exec-plan，保持文档与实际进度一致**
6. **输出结果应可直接复制到 Markdown、飞书、Notion 等工具**
