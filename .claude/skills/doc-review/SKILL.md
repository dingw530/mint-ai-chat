# Doc Review — 基于文档的任务产出审查

## 核心原则

审查的目的是**确保实现与文档一致**，既要发现"文档有但没做"的遗漏（少做），也要发现"做了但文档没规划"的偏离（多做）。审查报告仅做客观记录，不自动修改代码。

## 工作流程

### Step 1：确定审查范围

**定位源文档**（按优先级）：
1. 用户明确指定了 exec-plan 或 design-doc 路径 → 直接使用
2. 未指定时，检查 `docs/exec-plans/active/` 下是否有进行中的 exec-plan → 取最新的
3. 如 active 目录为空，检查 `docs/exec-plans/completed/` → 取最新的已归档 plan
4. 根据 exec-plan 头部 `关联设计文档` 字段找到对应的 design-doc

**定位代码变更**：
- 优先使用 `git diff` 或 `git log` 查看当前分支的变更
- 如果无 git 信息，询问用户变更了哪些文件
- 读取新增和修改的文件内容，了解实际实现

### Step 2：解析文档预期

#### 从 exec-plan 中提取：

| 提取项 | 说明 |
|--------|------|
| TP 列表 | 各 TP 的描述、关联 DS/FP/AC、预期产出文件 |
| 完成定义 | 文档头部的 checkboxes，代表最上层的验收标准 |
| 影响范围 | "本次范围" / "影响范围"中提及的新增/修改文件 |
| 追溯总览 | TP 与需求的关联矩阵 |
| 执行记录 | 各 TP 的实际执行状态和产出记录 |

#### 从 design-doc 中提取：

| 提取项 | 说明 |
|--------|------|
| DS 章节 | 详细设计中的接口签名、类结构、DB schema |
| 接口契约 | API-xxx 定义的请求/响应格式、端点 |
| 影响范围 | "影响与风险"中列出的新增/修改文件清单 |
| 验证标准 | 发布与验证章节的 AC-xxx 验收标准列表 |
| 最终决策 | 选定的方案和明确不选的方案 |

### Step 3：扫描实际代码产出

执行以下检查：
1. **文件清单对照** — 文档预期的文件 vs 实际新增/修改的文件
2. **代码结构对照** — 文档定义的类/方法/接口 vs 实际代码中的实现
3. **API 契约对照** — 文档定义的端点/请求/响应 vs 实际路由代码
4. **DB schema 对照** — 文档定义的表/字段 vs 实际 migration/DB 代码
5. **验收标准逐条核对** — 检查每条 AC 是否在代码中可验证

### Step 4：生成审查报告

按照以下结构生成报告，并**追加到 exec-plan 文件的 `## 执行记录` 末尾**（添加 `## 终审报告（doc-review）` 章节）：

```
## 终审报告（doc-review）

### 审查概要
- 审查日期：{YYYY-MM-DD HH:mm}
- 源文档：{exec-plan 路径}、{design-doc 路径}
- 审查范围：全部 {N} 个 TP
- 审查方式：{git diff / 文件扫描}

### TP 逐项审查

对每个 TP，按以下格式报告：

#### TP-NNN：{标题}
| 维度 | 结果 |
|------|------|
| 预期产出 | {文档要求的文件/功能} |
| 实际产出 | {实际产生的文件/功能} |
| 差异判定 | ✅ 完全匹配 / ⚠️ 部分匹配 / ❌ 未匹配 |

差异详情：
- ✅ {实现与文档一致的内容}
- ❌ **少做** — {文档有但未实现的内容}
- ⚠️ **多做** — {实现了但文档未规划的内容}
- ⚠️ **偏差** — {实现了但与文档描述不一致的内容}

---

### 验收标准核对

| 验收标准 | 状态 | 说明 |
|----------|------|------|
| AC-XXX | ✅/❌/⚠️ | {实现情况说明} |
| ... | ... | ... |

### 整体结论

- ✅ **通过**：全部验收标准满足，实现与文档一致
- ⚠️ **有条件通过**：存在 P2 以下问题，不影响核心功能（列出需跟进项）
- ❌ **不通过**：存在 P0/P1 问题，需要修复

### 问题清单

| # | 严重度 | 描述 | 涉及TP | 建议 |
|---|--------|------|--------|------|
| 1 | P0 | ... | TP-NNN | ... |
```

### Step 5：写入文件

1. 读取 exec-plan 的完整内容
2. 在 `## 执行记录` 章节末尾插入审查报告
3. 如果 exec-plan 原来没有 `## 执行记录`，先创建该章节再加入报告
4. 更新 exec-plan 文档信息的"状态"字段为"已完成（已审查）"
5. 保存文件

## 严重度定义

| 级别 | 定义 | 示例 |
|------|------|------|
| **P0** | 核心功能缺失或错误，导致无法使用 | 关键 API 未实现、DB 表缺失、SSE 流不通 |
| **P1** | 重要功能偏差，影响体验但有绕行方案 | 前端状态不对、UI 逻辑与设计不符 |
| **P2** | 次要问题，不影响功能但影响质量 | CSS 样式偏差、日志格式不符 |
| **P3** | 建议性改进，非必须 | 代码组织优化、命名规范 |

## 注意事项

1. **确认变更已完成** — 生成报告前确认用户已完成所有代码变更（文件已保存、git 未暂存变更已确认）
2. **未完成 TP 处理** — 如果 exec-plan 中有状态为"待启动"或"进行中"的 TP，提示用户并询问是否继续审查
3. **客观记录** — 审查报告只记录差异和问题，不自动修复代码
4. **格式保持** — 插入审查报告时保持 exec-plan 原有格式，不破坏已有内容
5. **不存在 exec-plan 的处理** — 如果未找到 exec-plan，询问用户是否仍要进行审查（此时仅对照 design-doc）

## 示例

### 用户请求审查

> 用户：帮我审查一下智能路由这个任务的产出

Skill 行为：
1. 自动定位 `docs/exec-plans/completed/2026-05-05-intelligent-routing-exec-plan.md`
2. 根据 exec-plan 找到关联的 `docs/design-docs/2026-05-05-intelligent-routing-design-doc.md`
3. 扫描代码变更（git diff 或文件扫描）
4. 逐 TP 对照，逐个 AC 核对
5. 将审查报告追加到 exec-plan 的执行记录
6. 告知用户审查结论

### 用户指定文件

> 用户：审查一下这个任务的产出，exec-plan 在 docs/exec-plans/active/xxx.md

Skill 行为：
1. 直接使用用户指定的 exec-plan 路径
2. 后续流程同上
